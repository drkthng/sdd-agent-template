# Spec-Driven Development (SDD) Agent Template

A project scaffolding template designed to enforce strict engineering discipline in AI-assisted coding environments. 

This framework was specifically developed to mitigate the limitations of AI coding agents in IDEs like **Antigravity** (and applies equally to Cursor, Claude Code, or generic CLI agents). 

## The Problem: AI "Vibe Coding" Limitations

Modern AI coding agents are incredibly fast, but they suffer from predictable failure modes during complex project development:

1. **Context Rot:** As the session lengthens, the agent forgets architectural decisions, hallucinates file paths, and degrades the codebase.
2. **Lack of Autonomous Agent Spawning:** In environments like Antigravity, agents cannot autonomously spawn sub-agents to clear their context window. They will try to push through a degraded context until the project breaks.
3. **The "Output as Verification" Fallacy:** Agents frequently write 500 lines of code, read their own output, and confidently declare the task "complete" without ever executing the compiler or test suite.
4. **Static Rule Bloat:** Providing an agent with hardcoded rules for every possible framework exhausts the context window before coding even begins.

## The Solution: Spec-Driven Development (SDD)

This template enforces a strict **Plan → Execute → Verify** loop through system prompts and externalized memory files. 

### Core Mechanisms

* **Externalized Memory (`PROGRESS.md` & `OBSTACLES.md`):** Agents are instructed to treat these files as their source of truth, bypassing the unreliability of conversational history. When a user must manually spawn a fresh agent in Antigravity to clear context rot, the new agent simply reads these files to instantly resume work.
* **Forced Execution:** The system prompt strictly prohibits declaring a task complete without explicit usage of terminal execution tools or browser verification.
* **Micro-Stepping:** If Test-Driven Development (TDD) is infeasible, agents are restricted to one logical unit of work (one function/endpoint) before they must run a build check.
* **Dynamic Tech-Stack Generation:** Instead of shipping with 50 static framework rules, the agent scans `docs/SPEC.md` and `package.json` on startup. It then dynamically generates the optimal ruleset for that specific stack and saves it to `.agent/rules/tech-stacks/` for the remainder of the project.

## Project Structure

```text
.agent/                        # Agent instructions (Do not delete)
  system_prompt.md             # Master agent instructions & boundaries
  rules/                       # Behavioral rules loaded per-task
    tech-stacks/               # Auto-generated by the agent at runtime
  workflows/                   # Pre / during / post-task checklists
docs/                          # Project documentation (Source of Truth)
  SPEC.md                      # Architecture, requirements, tech stack
  PROGRESS.md                  # What has been done / what is next
  OBSTACLES.md                 # Known issues and exact solutions
scaffold-sdd.py                # Standalone Python script to generate this folder
